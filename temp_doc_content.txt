LLM Prompt Design Guidance
Since most identity fields are now pre-filled from the DB, the LLM prompt should be focused and lean. Structure the prompt as follows:

Prompt Section
Content
Context inject
Provide the pre-fetched accused list: name, accused_id, person_id, age, gender — so the LLM can match narrative references to actual accused records.
Task
For each accused person identified in the brief_facts, extract: (1) their specific role_in_crime, (2) key_details summary. Only extract fields that are flagged as missing in the input payload.
Fallback fields
If address is empty string, extract address from narrative. If age is null, extract age from narrative. If alias_name is empty, look for '@' name patterns in brief_facts.
Output format
Return structured JSON keyed by accused_id. This makes it easy to merge back with DB-fetched rows without positional matching.
Avoid asking LLM to re-extract
Do NOT ask LLM for name, gender, phone, seq_num, accused_type, is_ccl, status — these are reliably in the DB and re-extracting wastes tokens and introduces error risk.

Data Quality & Edge Case Notes

SKIP RULE: If accused.person_id IS NULL for a given accused row, skip that row entirely — the person is not a real accused and should not appear in brief_facts_accused.
ACCUSED CODE FALLBACK: If accused.accused_code is NULL or empty, the LLM must read brief_facts and assign A1, A2, A3… in order of first mention of each person's name. LLM matches names from persons.full_name to identify the correct accused_id.
persons.alias may contain the full 'name @alias' format — strip to just the alias portion when populating alias_name.
Some persons rows have no present_* address fields (e.g. only nationality = 'India') — treat as empty and trigger LLM fallback.
accused.accused_code (A1, A2...) maps to bf_accused.person_code — do not confuse with persons.person_id.
accused.type ('Accused' / 'Known') maps to bf_accused.accused_type — no transformation needed.
accused.accused_status is the long-form status (e.g. 'Arrest By Police and Remanded') — store as-is in bf_accused.status.
phone_numbers in bf_accused may be an array/JSON column — wrap persons.phone_number in an array if needed.
existing_accused flag: set True if the accused_id already exists in the DB before this ETL run (detect via prior lookup).
source_*_fields JSON columns should record which fields came from DB vs LLM for full auditability.
